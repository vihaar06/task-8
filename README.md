# E-Commerce Order Analytics System

Intern mini project: build a small end-to-end pipeline (Python + SQL) that
generates messy e-commerce data, cleans it, loads it into SQLite, runs
16 analytical SQL queries, and serves a CLI reporting tool.

## Project structure

```
ecommerceAnalytics/
├── generateData.py          Part 1 - generates the 4 raw CSVs with intentional issues
├── cleanData.py              Part 2 - cleaning functions + data quality report
├── loadDatabase.py           loads cleaned CSVs into ecommerce.db (SQLite)
├── reportCli.py               Part 4 - CLI reporting tool (sqlite3 only)
├── edgeCaseTests.py          Part 5 - edge case test functions
├── sql/
│   ├── schema.sql            table definitions (PK/FK/CHECK constraints, indexes)
│   └── analysisQueries.sql   Part 3 - the 16 required SQL queries
├── data/
│   ├── raw/                  generated, messy CSVs
│   └── cleaned/               cleaned CSVs (output of cleanData.py)
├── output/
│   ├── dataQualityReport.txt generated by cleanData.py
│   └── sampleReports/        example query output
├── ecommerce.db               SQLite database (created by loadDatabase.py)
└── README.md
```

## How to run

```bash
pip install pandas

# Part 1: generate raw data (data/raw/*.csv)
python3 generateData.py

# Part 2: clean it (data/cleaned/*.csv + output/dataQualityReport.txt)
python3 cleanData.py

# Load cleaned data into SQLite
python3 loadDatabase.py

# Part 3: run the analysis queries against ecommerce.db, e.g.
sqlite3 ecommerce.db < sql/analysisQueries.sql
# (or open ecommerce.db in any SQLite client / DB Browser for SQLite)

# Part 4: interactive CLI report
python3 reportCli.py

# Part 5: edge case tests
python3 edgeCaseTests.py
```

## Part 1 - Data generation

`generateData.py` produces:

| File              | Rows  | Intentional issues |
|--------------------|------:|---------------------|
| `customers.csv`    | 600   | ~2% invalid emails (missing `@` or domain) |
| `products.csv`     | 520   | ~20% of product names have extra whitespace / inconsistent casing |
| `orders.csv`       | 1500  | ~5% missing `customer_id`; ~8% dates written as `DD-MM-YYYY` instead of `YYYY-MM-DD HH:MM:SS` |
| `order_items.csv`  | ~2700 | ~3% negative `quantity` (returns); 8 rows reference `order_id`s that don't exist in `orders.csv` |

Referential integrity is preserved for the "happy path" — every `order_id`,
`customer_id`, and `product_id` used is drawn from the pool actually
generated in the other tables, except for the small, deliberate batch of
orphaned `order_items` rows used to exercise `checkReferentialIntegrity()`.

## Part 2 - Cleaning (`cleanData.py`)

- `cleanOrders(df)` — parses `order_date` (accepting both the canonical
  format and the `DD-MM-YYYY` variant) and fills missing `customer_id`
  with the sentinel `'UNKNOWN'`.
- `cleanProducts(df)` — trims whitespace and title-cases `product_name`.
- `validateEmails(df)` — returns the list of `customer_id`s with an
  invalid email (no `@`, or no usable domain after it).
- `checkReferentialIntegrity(ordersDf, orderItemsDf)` — returns the
  `order_items` rows whose `order_id` has no matching row in `orders`;
  these rows are excluded before loading into SQLite.

Running it prints and saves `output/dataQualityReport.txt`, e.g.:

```
orders.csv
  - rows with a date originally in DD-MM-YYYY format (fixed): 116
  - rows with missing customer_id (set to 'UNKNOWN'):         73
products.csv
  - rows with messy product_name (trimmed / title-cased): 101
customers.csv
  - customer_ids with invalid emails: 11
order_items.csv
  - rows referencing a non-existent order_id (dropped): 8
```

## Part 3 - SQL analysis (`sql/analysisQueries.sql`)

Covers all 16 required queries: revenue per category, top customers,
monthly order counts, delivery/return diagnostics, running totals
(window functions), `DENSE_RANK` product ranking, `LAG` gap analysis,
multi-level CTEs, `NTILE` quartile segmentation, YoY comparison,
`FIRST_VALUE`/`LAST_VALUE` category-shift detection, cumulative revenue
distribution, cohort/retention analysis, and a self-join "frequently
bought together" query.

## Part 4 - CLI reporting tool (`reportCli.py`)

Uses only the `sqlite3` standard-library module. Prompts for a report
type (`daily`/`weekly`/`monthly`) and a date range, then prints total
orders/revenue/unique customers, the top 3 products by revenue, and a
percentage comparison against the immediately preceding period of equal
length.

## Part 5 - Edge case handling (`edgeCaseTests.py`)

Four test functions, each with an assertion, covering:

1. An `order_items` row whose `order_id` doesn't exist in `orders`.
2. `discount_percent > 100`.
3. `quantity == 0`.
4. `order_date` in the future.

Run with `python3 edgeCaseTests.py`.
